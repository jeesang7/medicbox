Purpose

The X-CUBE-AWS (STM32 AWS IoT software expansion for STM32Cube)

  • provides a qualified port of the Amazon FreeRTOS operating system to the supported boards;
  • offloads - wherever available - the security-critical operations to the on-board STSAFE-A110 Secure Element, during:
    • the MCU boot process,
    • the TLS device authentication towards the AWS IoT Core server,
    • the verification of the Over-the-air (OTA) update firmware image integrity and authenticity.

v2.x follows the v1.x versions. It connects to the same AWS IoT Core Services and is compatible with the same cloud services.

But it is a different thing: It is based on the complete FreeRTOS software distribution which coexists with the STM32 Cube environment, and not on the AWS IoT Device SDK for Embedded C single middleware library anymore.

Both the aws_demos and the aws_tests reference applications of FreeRTOS are provided.

Known Limitations

  • The compilation warnings of the bootloader applications are due to the disabled security features, and to the mbed-crypto third party library.

  • The compilation warnings of the user applications are due to the FreeRTOS code or to to its dependencies
    • EWARM
      Pa093: implicit conversion from floating point to integer
  Pe177: variable "" was declared but never referenced
  Pe188: enumerated type mixed with another type
  Pe223: function "" declared implicitly
  Pe236: controlling expression is constant
  Pe546: transfer of control bypasses initialization
  Pe550: variable "" was set but never used
    • STM32CubeIDE
      -Wunused-variable
  -Wunused-const-variable=
  -Wimplicit-function-declaration
  -Wformat=

  • One of the FreeRTOS Qualification test cases, Generate_MD5Hash, is failed with IAR 8.32.3 in High compiler optimization level.
    Reducing the optimization level (to Medium for iot_device_metrics.c) or upgrading IAR solves the issue.

  • Two of the FreeRTOS qualification test cases AFQP_SOCKETS_Threadsafe_SameSocketDifferentTasks and AFQP_SECURE_SOCKETS_Threadsafe_SameSocketDifferentTasks are known to occasionally fail. In that case, the next connections in the same application run generally fail too.
    For more details, please see the FreeRTOS Qualification Errata.
    In order to get a complete test status, the user may want to comment out both those test cases in iot_test_tcp.c::TEST_GROUP_RUNNER( Full_TCP ).

  • Passing the FreeRTOS qualification requires to ignore some socket errors caught in SOCKETS_Recv():
    • The explicit TLS remote disconnections must be handled as a receive underflow (AFQP_SECURE_SOCKETS_TwoSecureConnections);
    • The socket receive timeouts must be handled as a connection loss (AFQP_SECURE_SOCKETS_SetSockOpt_RCVTIMEO, AFQP_SECURE_SOCKETS_NonBlocking_Test, LastWillAndTestament, MQTT_Agent_MultiTaskTest).

    This can have unwanted effects for the user application.

  • In order to satisfy the FreeRTOS OTA update protocol, the provided secure bootloader initializes a 32 seconds reset watchdog that the user application must periodically renew.
    If the device cannot connect to the configured Wi-Fi access point within this time span (this is a blocking call), the device reboots.
    This reboot can interfere with the OTA update state flow of some test cases.

  • Running the post-build step that creates a pre-integrated binary of the the secure bootloader and the user application is not supported on Linux.
    [Not recommended] A possible workaround, with no easy debugger attachment, is to program the bootloader to the board, and install the user application
    • either through the Y-Modem protocol over ST-Link,
    • or through OTA update.

    • The build message “Generating the global elf file SBSFU and userApp” must be printed before the flash loader launch.
    • This is worth checking to be sure that the updated boot loader is used.

Bug fixes

  • eS-WiFi driver (B-L475E-IOT01A, 32F413H-DISCO): Solved random network disconnections that were due to a de-synchronization between the SPI and the GPIO.

  • TCP/IP configuration (32F769I-DISCO)
    • Generate a unique MAC address derived from the CPU ID in order to avoid address collisions when several F7 boards are connected to the same LAN.
    • Activate the local port number true randomization, to avoid possible TLS session confusions on server side after a board reset.
    • Activate the TCP keep-alive. Otherwise, the remote host may disconnect earlier than the first MQTT keep-alive message.

  • Improved error management

Other changes

  • Introduction of an internal network socket abstraction layer: NET API.

  • Source tree update to isolate the files which are common to all Cloud Cube packages.

  • RFU
    • Complies to HTTP 1.1.
    • Inter-operable with the Apache web server.
    • Supports server-side connection close, and client reconnection.
    • Supports HTTPS (the root CA certificate must be provisioned together with the Cloud and time servers’).
  • RTC initialization from HTTP
    • Network load reduction: HEAD command instead of GET .
    • Supports HTTPS (the root CA certificate must be provisioned together with the Cloud and RFU servers’. See the application readme.txt for more details).
